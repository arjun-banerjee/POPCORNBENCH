import torch
import torch.nn as nn

class Model(nn.Module):
    """
    Performs a depthwise-separable 2D convolution operation.

    Args:
        in_channels (int): Number of channels in the input tensor.
        out_channels (int): Number of channels produced by the convolution.
        kernel_size (int): Size of the convolution kernel.
        stride (int, optional): Stride of the convolution. Defaults to 1.
        padding (int, optional): Padding applied to the input. Defaults to 0.
        dilation (int, optional): Spacing between kernel elements. Defaults to 1.
        bias (bool, optional): If `True`, adds a learnable bias to the output. Defaults to `False`.
    """
    def __init__(self, in_channels: int, out_channels: int, kernel_size: int, stride: int = 1, padding: int = 0, dilation: int = 1, bias: bool = False):
        super(Model, self).__init__()
        self.depthwise = nn.Conv2d(in_channels, in_channels, kernel_size, stride=stride, padding=padding, dilation=dilation, groups=in_channels, bias=bias)
        self.pointwise = nn.Conv2d(in_channels, out_channels, kernel_size=1, bias=bias)
        
    def forward(self, x: torch.Tensor) -> torch.Tensor:
        """
        Performs the depthwise-separable 2D convolution.

        Args:
            x (torch.Tensor): Input tensor of shape (batch_size, in_channels, height, width).

        Returns:
            torch.Tensor: Output tensor of shape (batch_size, out_channels, height_out, width_out).
        """
        x = self.depthwise(x)
        x = self.pointwise(x)
        return x

We need to write a custom CUDA kernel that replaces the forward computation of a depthwise-separable 2D convolution: depthwise conv (groups=in_channels) followed by pointwise conv (1x1). The goal is speedup over PyTorch reference. First, let's understand what the model does: - Input: [B, C_in, H, W] - Depthwise conv: Conv2d with groups=C_in, kernel_size=K, stride=S, padding=P, dilation=D, bias maybe. Output: [B, C_in, H_out, W_out] where H_out = floor((H + 2*P - D*(K-1) - 1)/S) + 1. - Pointwise conv: Conv2d 1x1, stride=1, padding=0, same bias flag. Output: [B, C_out, H_out, W_out]. We need to fuse these two operations into a single kernel to avoid intermediate memory reads/writes and to maximize throughput. Given the test parameters: B=16, C_in=64, C_out=128, K=3, H=W=512, S=1, P=1, D=1, bias=False. So the depthwise conv is 3x3 with padding 1, stride 1, dilation 1. Output size remains 512x512. Then pointwise conv is 1x1 from 64 to 128 channels. The naive approach: launch a fused kernel where each thread computes one output pixel (or a tile) for all output channels. Given C_in=64, C_out=128, K=3, H=W=512, B=16. Total output elements: 16 * 128 * 512 * 512 = 536,870,912. If each thread computes one output element, we need ~537M threads, which is a lot. In CUDA, we can have up to ~2^31 threads per grid dimension? Actually max grid x dimension is 2^31-1. So it's okay. But each thread would compute a dot product over C_in * K * K + C_in (pointwise) operations. But wait, the fused kernel can compute output[B, oc, h, w] = sum_{ic, ky, kx} weight_dw[ic, ky, kx] * weight_pw[oc, ic] * x[B, ic, h*stride+ky*D-P, w*stride+kx*D-P] + bias terms. Since it's depthwise, each input channel has its own spatial filter. So the fused weight for each output channel oc and input spatial position is weight_pw[oc, ic] * weight_dw[ic, ky, kx]. So we can precompute a combined weight of shape [C_out, C_in, K, K] = 128*64*3*3 = 73,728 floats. That's tiny. Then the operation is just a standard convolution with this combined weight (groups=1, kernel KxK). Wait, is that correct? Let's verify: depthwise conv output y[ic, h, w] = sum_{ky,kx} w_dw[ic, ky, kx] * x[ic, h_s, w_s]. Then pointwise conv output z[oc, h, w] = sum_{ic} w_pw[oc, ic] * y[ic, h, w]. So z[oc, h, w] = sum_{ic, ky, kx} w_pw[oc, ic] * w_dw[ic, ky, kx] * x[ic, h_s, w_s]. Yes, that's exactly a standard convolution with weight w_comb[oc, ic, ky, kx] = w_pw[oc, ic] * w_dw[ic, ky, kx]. And bias: if bias is True, then depthwise has bias_dw[ic] and pointwise has bias_pw[oc]. The fused bias would be bias_pw[oc] + sum_{ic} w_pw[oc, ic] * bias_dw[ic]? Wait, no. Let's trace: y[ic, h, w] = sum_{ky,kx} w_dw[ic, ky, kx] * x[ic, h_s, w_s] + b_dw[ic]. z[oc, h, w] = sum_{ic} w_pw[oc, ic] * y[ic, h, w] + b_pw[oc] = sum_{ic} w_pw[oc, ic] * (sum_{ky,kx} w_dw[ic, ky, kx] * x[ic, h_s, w_s] + b_dw[ic]) + b_pw[oc] = sum_{ic, ky,kx} w_pw[oc, ic] * w_dw[ic, ky, kx] * x[ic, h_s, w_s] + sum_{ic} w_pw[oc, ic] * b_dw[ic] + b_pw[oc]. So the fused bias is b_comb[oc] = b_pw[oc] + sum_{ic} w_pw[oc, ic] * b_dw[ic]. We can precompute the combined weight and bias in the constructor or in the first forward call. Then the operation reduces to a standard conv2d with groups=1, kernel_size=K, stride=S, padding=P, dilation=D, in_channels=C_in, out_channels=C_out. But the standard PyTorch conv2d for these sizes might already be highly optimized (cuDNN). However, PyTorch uses two separate conv calls with intermediate memory. Fusing them into a single custom kernel may be faster because we eliminate the intermediate tensor and reduce memory bandwidth. But cuDNN might also fuse them implicitly? Usually not; depthwise separable is two separate cuDNN calls. A custom fused kernel can beat it, especially if we can exploit shared memory or keep everything in registers. However, writing a generic conv2d kernel that matches cuDNN performance is hard. But given the sizes, maybe a simple kernel that loads input into shared memory and computes the output can be competitive because of the fusion and because cuDNN overhead for two calls.
