import torch
import torch.nn as nn


class Model(nn.Module):
    """
    Scatter-add edge features into destination-node accumulators.

    This is a canonical graph primitive for COO-formatted workloads where edge
    messages are already materialized and must be reduced into node states.
    """

    def __init__(self, num_nodes):
        super().__init__()
        self.num_nodes = num_nodes

    def forward(
        self,
        dst_idx: torch.Tensor,
        edge_feat: torch.Tensor,
    ) -> torch.Tensor:
        feat_dim = edge_feat.shape[1]
        out = torch.zeros(self.num_nodes, feat_dim, dtype=edge_feat.dtype, device=edge_feat.device)
        out.index_add_(0, dst_idx.long(), edge_feat)
        return out


num_nodes = 768
num_edges = 16384
feat_dim = 64

    edge_feat = torch.randn(num_edges, feat_dim, dtype=torch.float32)
    return [dst_idx, edge_feat]

We need to write a custom CUDA kernel for scatter-add of edge features into node accumulators. We have reference: out = torch.zeros(num_nodes, feat_dim, dtype=edge_feat.dtype, device=edge_feat.device) out.index_add_(0, dst_idx.long(), edge_feat) We need to write a custom kernel that does this faster. We need to know GPU specs. Let's call get_gpu_specs() first to understand target hardware. Then write kernel, compile, run correctness, optimize. Let's inspect reference operation: It's essentially computing for each edge e, destination node d=dst_idx[e], adding edge_feat[e, :] to out[d, :]. So a gathering via index_add. We need to exploit parallelism. Since num_edges=16384, num_nodes=768, feat_dim=64. Torch's index_add_ is usually implemented with an atomicAdd per feature per edge. Since there are many edges mapping to same destination node, there is contention. Custom kernel can exploit shared memory per block, or sort and reduce, or use block-level reductions. Since num_nodes=768 is small, we might assign one block per node, or per feature dimension. But let's see. We can do a simple kernel where each thread processes one edge and one feature, doing atomicAdd on global memory. But atomicAdd contention on same node may be high if many edges map to same node. But 16384 edges over 768 nodes => average ~21 edges per node. Contention is moderate. We can maybe use shared memory per block: block processes a subset of edges, accumulates into a shared memory array of shape (nodes_in_block, feat_dim), then writes back. But nodes per block are not known. Alternative: sort edges by destination node and then do a simple parallel reduction per node. Given small num_nodes=768 and feat_dim=64, we can process each node with one block: each block loads all edges (or a chunk) and checks if dst matches node id, then adds edge_feat to a shared accumulator per feature. This is O(num_nodes * num_edges) work done in parallel across blocks, but each block does full scan of edges. That's 768*16384 = 12.5M checks; each check is simple. Given we have many SMs, that might be okay, but maybe not optimal. Actually, 768 blocks each scanning 16384 edges => 12.5M thread iterations. With enough threads per block (e.g., 256), that's 768*64 = 49152 threads. Each thread does ~256 iterations. That's maybe okay but memory bandwidth bound because each block reads entire edge_feat array (1MB) => 768 * 1MB = 768 MB read, which is high. Could be okay depending on bandwidth but not ideal. Better approach: assign each edge to a thread (or warp) and do atomicAdd into output. Because num_edges=16384, each thread processes one edge and loops over feat_dim (64). So total work: 16384*64 = ~1M operations. Memory reads: reading edge_feat (1MB) and dst_idx (64KB). Writes: 768*64=49KB output. So it's memory bandwidth bound but with minimal work. Let's consider atomicAdd approach: each thread reads dst_idx[e], then for each f in 0..63, atomicAdd(&out[dst*64 + f], edge_feat[e*64 + f]). But atomicAdd on float32 is relatively slow. However with only ~21 collisions per node on average, contention is low. This might be close to peak. Alternative: each warp processes a node? No, easy is just do coalesced loads per edge.
