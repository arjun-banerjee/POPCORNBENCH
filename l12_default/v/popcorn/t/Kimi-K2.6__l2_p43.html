import torch
import torch.nn as nn
import torch.nn.functional as F


class Model(nn.Module):
    """
    Contact map prediction head.  Takes residue-pair embeddings and
    predicts a symmetric binary contact probability matrix (residue i
    within 8 Å of residue j).  Uses a small MLP on the symmetrised pair
    representation followed by a sigmoid.  Common final stage in protein
    structure prediction pipelines.
    """

    def __init__(self, pair_dim, hidden_dim):
        super().__init__()
        self.layer_norm = nn.LayerNorm(pair_dim)
        self.mlp = nn.Sequential(
            nn.Linear(pair_dim, hidden_dim),
            nn.ReLU(),
            nn.Linear(hidden_dim, hidden_dim),
            nn.ReLU(),
            nn.Linear(hidden_dim, 1),
        )

    def forward(self, pair_repr: torch.Tensor) -> torch.Tensor:
        """
        Args:
            pair_repr: (B, N, N, pair_dim)
        Returns:
            contact_probs: (B, N, N) – symmetric contact probability
        """
        x = self.layer_norm(pair_repr)
        # Symmetrise
        x = (x + x.permute(0, 2, 1, 3)) / 2.0
        logits = self.mlp(x).squeeze(-1)  # (B, N, N)
        return torch.sigmoid(logits)


pair_dim = 64
hidden_dim = 32
seq_len = 64
batch_size = 4


def get_inputs():
    return [torch.randn(batch_size, seq_len, seq_len, pair_dim)]


def get_init_inputs():
    return [pair_dim, hidden_dim]

We need to write a custom CUDA kernel that replicates this model: layer norm over pair_dim, symmetrize by averaging with transpose over dims 1 and 2, then MLP: linear -> ReLU -> linear -> ReLU -> linear -> squeeze(-1) -> sigmoid. Given small dimensions: pair_dim=64, hidden_dim=32, seq_len=64, batch=4. The reference is straightforward PyTorch operations, which will be already fairly optimized on GPU via cuDNN/cuBLAS. But we can try to fuse everything into a single kernel to reduce memory traffic and synchronization overhead. Key challenge: small seq_len=64 and hidden_dim=32. The model is basically a bunch of elementwise and small matmuls. Since pair_dim=64 and hidden_dim=32 and N=64, each linear is (N*N) x pair_dim -> (N*N) x hidden_dim. With batch=4, total M=4*64*64=16384. That's enough for cuBLAS to be efficient, but still, fusing the entire forward pass into one kernel might reduce the number of passes over memory.
