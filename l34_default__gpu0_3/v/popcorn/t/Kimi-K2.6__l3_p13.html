import torch
import torch.nn as nn
import torch.nn.functional as F


class Model(nn.Module):
    """
    Conditional Variational Autoencoder (CVAE) with full encoder–decoder
    forward pass and loss computation.  The encoder conditions on both
    input x and label y; the decoder conditions on z and y.  Returns
    reconstruction loss + KL divergence.
    """

    def __init__(self, input_dim, label_dim, hidden_dim, latent_dim):
        super().__init__()
        # Encoder: q(z | x, y)
        self.enc = nn.Sequential(
            nn.Linear(input_dim + label_dim, hidden_dim),
            nn.ReLU(),
        )
        self.enc_mu = nn.Linear(hidden_dim, latent_dim)
        self.enc_logvar = nn.Linear(hidden_dim, latent_dim)

        # Decoder: p(x | z, y)
        self.dec = nn.Sequential(
            nn.Linear(latent_dim + label_dim, hidden_dim),
            nn.ReLU(),
            nn.Linear(hidden_dim, input_dim),
        )

    def forward(self, x: torch.Tensor, y: torch.Tensor) -> tuple:
        """
        Args:
            x: (B, input_dim)  – input data
            y: (B, label_dim)  – conditioning label (one-hot)
        Returns:
            (loss, recon_loss, kl_loss):
                loss:       scalar – total CVAE loss
                recon_loss: scalar
                kl_loss:    scalar
        """
        h = self.enc(torch.cat([x, y], dim=-1))
        mu = self.enc_mu(h)
        logvar = self.enc_logvar(h)

        std = torch.exp(0.5 * logvar)
        z = mu + std * torch.randn_like(std)

        x_recon = self.dec(torch.cat([z, y], dim=-1))
        recon_loss = F.mse_loss(x_recon, x, reduction="sum") / x.shape[0]
        kl_loss = -0.5 * (1 + logvar - mu.pow(2) - logvar.exp()).sum() / x.shape[0]
        loss = recon_loss + kl_loss
        return loss, recon_loss, kl_loss


input_dim = 256
label_dim = 10
hidden_dim = 128
latent_dim = 32
batch_size = 128


def get_inputs():
    x = torch.randn(batch_size, input_dim)
    y = torch.zeros(batch_size, label_dim)
    y.scatter_(1, torch.randint(0, label_dim, (batch_size, 1)), 1.0)
    return [x, y]


def get_init_inputs():
    return [input_dim, label_dim, hidden_dim, latent_dim]

We need to write a custom CUDA kernel that replaces the forward computation of a small CVAE model. The model consists of: 1. Concatenate x and y -> (B, input_dim + label_dim) 2. Linear -> hidden_dim, ReLU 3. Linear -> mu (latent_dim) 4. Linear -> logvar (latent_dim) 5. Sample z = mu + exp(0.5 * logvar) * eps 6. Concatenate z and y -> (B, latent_dim + label_dim) 7. Linear -> hidden_dim, ReLU 8. Linear -> input_dim 9. Compute MSE loss (sum / B) 10. Compute KL loss (sum / B) The key optimization is to fuse all these operations into a single CUDA kernel. Since the model is small and fully connected, a straightforward fused kernel is likely the fastest. We need to handle arbitrary batch sizes. The weight dimensions are fixed at model init. Let's inspect the dimensions: - x: (B, 256) - y: (B, 10) - concat(x, y): (B, 266) - enc1 weight: (128, 266), bias: (128) - enc1 out: (B, 128) - enc_mu weight: (32, 128), bias: (32) - enc_logvar weight: (32, 128), bias: (32) - z: (B, 32) - concat(z, y): (B, 42) - dec1 weight: (128, 42), bias: (128) - dec2 weight: (256, 128), bias: (256) We can implement a single kernel that, for each sample in the batch: 1. Loads x and y into shared memory or registers. 2. Computes h = ReLU(W_enc * [x;y] + b_enc). This is a matvec of size 128. 3. Computes mu = W_mu * h + b_mu (size 32). 4. Computes logvar = W_logvar * h + b_logvar (size 32). 5. Samples z. 6. Computes h2 = ReLU(W_dec1 * [z;y] + b_dec1) (size 128). 7. Computes x_recon = W_dec2 * h2 + b_dec2 (size 256). 8. Computes per-sample recon loss and kl loss, stores to global memory or computes final sum with atomics/parallel reduction. Wait, if we launch one thread per sample, the matrix-vector multiplications are small. For B=128, launching 128 blocks of e.g. 128 threads each, one block per sample, could be okay. Or one thread per sample and do everything sequentially? No, 128 threads doing 128x266 matvec sequentially is too much work per thread and won't use the GPU well. Better approach: Launch one block per sample (B blocks). Within each block, use multiple threads to cooperatively compute the matrix-vector products. Or, since B is 128, and each sample is independent, we can use one warp (32 threads) per sample. Total warps = 128. SMs likely have enough capacity. For B up to e.g. 1024 or more, we'd want to be careful. Let's think about the workload per sample: - 1 matvec: 128 x 266. That's ~34K mul-adds. - 2 matvecs: 32 x 128 each. That's ~8K mul-adds total. - 1 matvec: 128 x 42. That's ~5K mul-adds. - 1 matvec: 256 x 128. That's ~33K mul-adds. Total: ~80K FLOPs per sample. For B=128, total ~10M FLOPs. Memory traffic: weights are ~ (128*266 + 32*128 + 32*128 + 128*42 + 256*128) = 34048 + 4096 + 4096 + 5376 + 32768 = 80K floats = 320KB. Inputs/outputs are small. This is memory-bound. So the best optimization is to fuse everything to avoid intermediate reads/writes to global memory, and ideally keep weights in shared memory or L1. Since weights are only 320KB, we can load them all into shared memory, or at least cache them well. However, different matvecs use different weights. We can process the weights in tiles. Let's design the kernel. Data layout: Weights are stored transposed or not? PyTorch uses (out_features, in_features). For matvec y = W * x + b, where W is (M, N) and x is (N). We can store W in row-major (M rows, N columns). Strategy 1: One thread block per batch element. Block size = 128 threads. Block processes one sample. Shared memory for activations: we need to store x (256), y (10), h (128), z (32), h2 (128), x_recon (256). That's about 682 floats = ~2.7KB. Shared memory for weights? We could load the current weight matrix tile into shared memory. But each matvec uses a different weight matrix. W_enc: (128, 266) -> 34048 floats -> 136KB. Too big for shared mem per block? Usually 48KB or 96KB max. So we can't load the whole W_enc into shared memory. We can tile it. Let's look at GPU specs.
