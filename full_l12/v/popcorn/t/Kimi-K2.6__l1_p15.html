import torch
import torch.nn as nn


class Model(nn.Module):
    """
    Scatter-add edge features into destination-node accumulators.

    This is a canonical graph primitive for COO-formatted workloads where edge
    messages are already materialized and must be reduced into node states.
    """

    def __init__(self, num_nodes):
        super().__init__()
        self.num_nodes = num_nodes

    def forward(
        self,
        dst_idx: torch.Tensor,
        edge_feat: torch.Tensor,
    ) -> torch.Tensor:
        feat_dim = edge_feat.shape[1]
        out = torch.zeros(self.num_nodes, feat_dim, dtype=edge_feat.dtype, device=edge_feat.device)
        out.index_add_(0, dst_idx.long(), edge_feat)
        return out


num_nodes = 768
num_edges = 16384
feat_dim = 64

reasoning
We need to implement a custom CUDA kernel for scatter-add edge features into destination-node accumulators. The reference uses `index_add_` (with index 0, i.e., dimension 0). `dst_idx` is shape (num_edges,). `edge_feat` is (num_edges, feat_dim). We want to produce out of shape (num_nodes, feat_dim) = zeros + index_add_ by dst_idx. Goal: faster than PyTorch reference. Let's think about optimization. We need to write a Python module that defines ModelNew. At module load time, use `torch.utils.cpp_extension.load_inline` to compile and bind CUDA source. We need to accept same constructor arguments: `num_nodes`. Implement forward with same signature: `dst_idx`, `edge_feat`. Produce equivalent output. Key considerations: - Data types: fp32, but may be used on float32 Tensors. The index is int32. Need to handle int32 indices (or long). In reference, dst_idx is int32 but `index_add_` expects LongTensor; they do `.long()`. Our kernel can accept int32 indices to avoid conversion, but inside we need to read them as int32. - Sizes: num_nodes = 768, num_edges = 16384, feat_dim = 64. This is relatively small. However, we can aim for high performance. The kernel design options: 1. Simple thread-per-edge or thread-per-element. Since feat_dim = 64, we could have each thread handle one edge feature element. But we need to atomically add to `out[dst, f]`. If we use thread-per-element, we need atomicAdd on float for each element. Since atomicAdd on global memory is okay. But there are 16384*64 = ~1M elements. On GPU, we can launch enough threads. 2. Use shared memory buffers per block. Since num_nodes = 768, we can have each block process a subset of edges, accumulate into shared memory buffers of size (num_nodes, feat_dim) = 768*64 = 49152 floats = 196608 bytes = ~192KB. Shared memory per SM is typically up to 164KB or more. Some GPUs have 100KB. 192KB might exceed typical shared memory limit (e.g., A100 has 164KB per SM). So cannot fit full num_nodes*feat_dim in shared memory per block unless using smaller block size or a subset of nodes. Could tile over nodes. Alternatively, we can have each block handle a contiguous chunk of the feature dimension. Since num_nodes*feat_dim = 768*64 = 49152. If we tile over nodes or features. For each block, allocate smem of size block_num_nodes * block_feat_dim. But because edges scatter to arbitrary dst nodes, each block would need to process only edges whose dst falls into the block's node tile. We would need to partition edges, which is expensive. Alternatively, we can use CUDA's `atomicAdd` but transpose the accumulation: each thread processes one feature element for one edge. Use 1D grid. Since atomicAdd on float for a small number of nodes may have contention. With 768 destination nodes and 16384 edges, average degree is 21.3. Contention is not huge. But with 64 features, each feature dimension is independent, so contention per output element is from all edges mapping to same node across same feature index. Max contention maybe up to ~30 edges. That's not bad. So a straightforward thread-per-element atomicAdd should be fine. But is it faster than PyTorch's `index_add_`? PyTorch's `index_add_` likely uses a similar kernel. However, we can try to improve by reading indices and data with vectorized loads (float4), using `atomicAdd` on floats. But wait: On Ampere and newer, there are `atomiAdd` for half, but not for float? Actually `atomicAdd` on float exists since early CUDA. So we can use `atomicAdd`. Another optimization: we can convert the problem into a matrix multiplication: If we create a sparse matrix (num_nodes x num_edges) with ones where edge maps to dst, then output = S * edge_feat. That is `torch.sparse.mm`. But overhead of constructing sparse matrix may be high. Better: custom kernel. Given small sizes, we might also use a sort-based approach: Sort edges by dst_idx, then each thread block processes a range of edges for the same dst node, and does a parallel reduction. This reduces atomic contention and improves memory locality. But with only ~1M elements, sorting overhead might not be worth. Could be a good approach. But if we want speedup, perhaps a simple kernel with good vectorization and efficient memory access is sufficient. Let's check GPU specs first.
